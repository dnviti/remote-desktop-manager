[~] TENANT-021 — Backend CRUD tenant + onboarding e migrazione utenti
------------------------------------------------------------------------------
  Priorita: CRITICA
  Dipendenze: TENANT-020
  Fase: 1 (Fondamenta)

  DESCRIZIONE:
  Implementare il layer backend completo per la gestione dei tenant:
  creazione, aggiornamento, eliminazione tenant. Implementare il flusso
  di onboarding: quando un utente crea un tenant, diventa OWNER.
  Gli utenti possono essere invitati nel tenant (via email o link).
  Creare endpoint per listare utenti del tenant e gestirne i ruoli.
  Utenti esistenti senza tenant possono crearne uno o essere invitati.

  DETTAGLI TECNICI:
  Backend — Layer completo:

    server/src/routes/tenant.routes.ts:
      POST   /api/tenants              → Crea tenant (l'utente diventa OWNER)
      GET    /api/tenants/mine         → Dettagli del tenant dell'utente
      PUT    /api/tenants/:id          → Aggiorna tenant (solo OWNER/ADMIN)
      DELETE /api/tenants/:id          → Elimina tenant (solo OWNER)
      GET    /api/tenants/:id/users    → Lista utenti del tenant
      POST   /api/tenants/:id/invite   → Invita utente (email)
      PUT    /api/tenants/:id/users/:userId  → Cambia ruolo utente
      DELETE /api/tenants/:id/users/:userId  → Rimuovi utente dal tenant

    server/src/controllers/tenant.controller.ts:
      - Validazione Zod per ogni endpoint
      - Verifica che l'utente appartenga al tenant (tranne creazione)

    server/src/services/tenant.service.ts:
      - createTenant(userId, name): crea tenant + imposta utente come OWNER
      - getTenant(tenantId): dettagli tenant
      - updateTenant(tenantId, data): aggiorna nome/slug
      - deleteTenant(tenantId): elimina con cascade (teams, members)
      - listTenantUsers(tenantId): lista utenti con ruolo
      - inviteUser(tenantId, email, role): cerca utente per email,
        se esiste lo aggiunge al tenant, altrimenti genera invito pendente
      - updateUserRole(tenantId, userId, role): cambia ruolo
      - removeUser(tenantId, userId): rimuovi da tenant e tutti i team

    server/src/middleware/tenant.middleware.ts:
      - requireTenant: middleware che verifica req.user.tenantId != null
      - requireTenantRole(minRole): verifica ruolo minimo (OWNER > ADMIN > MEMBER)
      - extractTenantContext: aggiunge tenantId al request per query scoping

  Modifica auth flow:
    - In auth.service.ts login(): includere tenantId e tenantRole nel
      payload JWT e nella risposta login
    - In AuthPayload: aggiungere tenantId?, tenantRole?
    - Utenti senza tenant: continuano a funzionare (backward compatible)

  Nota sicurezza:
    - Ogni query Prisma che accede a dati tenant-scoped DEVE filtrare
      per tenantId. Mai fidarsi solo dell'autenticazione JWT.
    - L'eliminazione dell'ultimo OWNER e' vietata

  FILE COINVOLTI:
