------------------------------------------------------------------------------
  Priorita: CRITICA
  Dipendenze: Nessuna

  DESCRIZIONE:

  viene salvata in una tabella pending. Quando il target sblocca il proprio

     - id: String @id @default(uuid())

     - tenantVaultKeyIV: String
     - tenantVaultKeyTag: String
     - distributorUserId: String (FK -> User)
     - createdAt: DateTime @default(now())

  3) Modificare initTenantVault() in secret.service.ts:

  4) Nuovo metodo processPendingDistributions(userId) in secret.service.ts:

[ ] VAULT-208 — Preservazione vault su password reset (Element-style, nessun wipe automatico)
------------------------------------------------------------------------------
  Priorita: CRITICA
  Dipendenze: Nessuna

  DESCRIZIONE:
  Quando un utente resetta la password senza recovery key, il sistema esegue
  un vault wipe completo (passwordReset.service.ts righe 246-337): tutte le
  credenziali connessioni, segreti vault, team/tenant vault keys vengono
  cancellati. Questo e' catastrofico per qualsiasi utente.

  Soluzione (stile Element/Matrix): NON cancellare MAI automaticamente i dati
  cifrati. Se l'utente non fornisce la recovery key durante il reset, i dati
  restano cifrati e il vault viene marcato come "in attesa di recovery".
  L'utente puo' fornire la recovery key in qualsiasi momento (sul form di
  reset password O nella sezione dedicata del Keychain). Se la recovery key
  e' persa definitivamente, l'utente puo' scegliere esplicitamente di
  resettare il vault (con warning e conferma esplicita). Stesso comportamento
  per account personali e membri tenant.

  intervallo). I risultati vengono trasmessi via Socket.IO per
  aggiornamenti in tempo reale. Rispetta i confini di rete: se una
  connessione ha un gateway assegnato, il probe viene effettuato
  attraverso il gateway (SSH probe via bastion, TCP diretto per guacd).

      - startMonitoring(connections[]): avvia probe periodici per le

      - stopMonitoring(userId): ferma i probe
      - probeConnection(connection): TCP probe verso host:port, timeout 5s
        - Se connessione ha gateway SSH_BASTION/MANAGED_SSH: eseguire
          il probe dal server (il target e' raggiungibile solo via gateway)

      - Broadcast stato a user room quando cambia
      - Client si connette al login, riceve aggiornamenti
    - Endpoint opzionale: POST /api/connections/:id/probe (probe singolo
      on-demand)

      (da aggiungere a Tenant o come setting separato)

        verde = online, rosso = offline, grigio = unknown/disabled
      - Tooltip con latenza e ultimo check
    - Creare client/src/hooks/useConnectionMonitor.ts:
      - Connessione al namespace Socket.IO /connection-monitor
      - Stato locale: Map<connectionId, { status, latencyMs, checkedAt }>
    - Estendere connectionsStore.ts o creare store dedicato:
      - connectionHealth: Record<string, HealthStatus>
      - updateConnectionHealth action
    - uiPreferencesStore: toggle per mostrare/nascondere indicatori
