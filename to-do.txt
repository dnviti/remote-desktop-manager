[ ] GATE-036 — Modello Gateway e CRUD backend (Prisma + API)
------------------------------------------------------------------------------
  Priorita: ALTA
  Dipendenze: TENANT-020 (completato), GUARD-026 (completato)

  DESCRIZIONE:
  Creare l'entita' Gateway nel database e le relative API CRUD. Un Gateway
  rappresenta un backend riutilizzabile (istanza guacd per RDP o bastion SSH)
  attraverso cui instradare le connessioni verso sottoreti/VLAN diverse.
  Casi d'uso: Proxmox con piu' SDN, datacenter multipli con subnet isolate,
  ambienti dove si aprono solo le porte verso guacd e il gateway SSH.
  I gateway sono tenant-scoped (ogni tenant gestisce i propri).

  DETTAGLI TECNICI:
  Backend:
    - Nuovo enum GatewayType { GUACD SSH_BASTION } in schema.prisma
    - Nuovo modello Gateway in schema.prisma:
      id                 String      @id @default(uuid())
      name               String      (nome visualizzato, es. "DC Milano", "Proxmox SDN-Lab")
      type               GatewayType (GUACD per istanze guacamole-daemon, SSH_BASTION per jump host)
      host               String      (hostname/IP del gateway)
      port               Int         (porta: 4822 per guacd, 22 per SSH bastion)
      description        String?     (descrizione opzionale)
      isDefault          Boolean     @default(false)  (un default per tipo per tenant)
      tenantId           String      (FK a Tenant, obbligatorio — isolamento tenant)
      createdById        String      (FK a User, chi ha creato il gateway)
      createdAt          DateTime    @default(now())
      updatedAt          DateTime    @updatedAt
      -- Credenziali cifrate per SSH_BASTION (nullable, non servono per GUACD):
      encryptedUsername   String?
      usernameIV         String?
      usernameTag        String?
      encryptedPassword  String?
      passwordIV         String?
      passwordTag        String?
    - Modello Connection: aggiungere gatewayId String? (FK opzionale a Gateway)
    - Migrazione Prisma (db:migrate)
    - Nuovi file backend: gateway.routes.ts, gateway.controller.ts, gateway.service.ts
    - Endpoint API:
      GET    /api/gateways           Lista gateway del tenant corrente
      POST   /api/gateways           Crea gateway (solo admin/owner tenant)
      PUT    /api/gateways/:id       Aggiorna gateway
      DELETE /api/gateways/:id       Elimina (fallisce se connessioni lo referenziano)
      POST   /api/gateways/:id/test  Test connettivita' (TCP connect a host:port)
    - Isolamento tenant: tutte le query filtrate per tenantId (pattern GUARD-026)
  File coinvolti:
    MODIFICARE: server/prisma/schema.prisma (Gateway, GatewayType, Connection.gatewayId)
    MODIFICARE: server/src/app.ts (montare gateway routes)
    CREARE:     server/src/routes/gateway.routes.ts
    CREARE:     server/src/controllers/gateway.controller.ts
    CREARE:     server/src/services/gateway.service.ts

  Dipendenze: GATE-036

  guacdHost e guacdPort nel token cifrato per ogni connessione.

      del token: { connection: { type: 'rdp', guacdHost, guacdPort, settings } }

      ha un gatewayId, caricare il Gateway e passare host/port del gateway
      a generateGuacamoleToken()
    - connection.service.ts: includere relazione gateway nelle query connessioni
    - Fallback: nessun gateway = guacd da config (GUACD_HOST/GUACD_PORT)

[ ] GATE-038 — Routing SSH tramite bastion/jump host
------------------------------------------------------------------------------
  Priorita: ALTA
  Dipendenze: GATE-036

  DESCRIZIONE:
  Instradare le connessioni SSH attraverso un bastion host (jump host) quando
  la connessione ha un gateway di tipo SSH_BASTION. Il server apre prima una
  connessione SSH verso il bastion, poi un tunnel TCP verso l'host target,
  e infine stabilisce la sessione SSH finale attraverso il tunnel.
  Se nessun gateway e' assegnato, connessione diretta (comportamento attuale).

  DETTAGLI TECNICI:
  Backend:
    - ssh.service.ts: nuova funzione createSshConnectionViaBastion() che:
      1. Connette al bastion host con credenziali del Gateway (decifrate dalla vault)
      2. Apre un canale TCP forwarded (client.forwardOut()) verso target host:port
      3. Crea un secondo SSH2 Client sullo stream forwarded verso il target
      4. Restituisce la sessione finale (stessa interfaccia SshSession)
    - ssh.handler.ts: controllare se la connessione ha un gateway SSH_BASTION;
      se si', usare createSshConnectionViaBastion() al posto di createSshConnection()
    - SFTP: funziona automaticamente (stesso client SSH2 sottostante)
    - Gestione cleanup: chiudere entrambi i client SSH (bastion + target)
      quando la sessione termina
    - Fallback: nessun gateway = connessione diretta (createSshConnection)
  File coinvolti:
    MODIFICARE: server/src/services/ssh.service.ts
    MODIFICARE: server/src/socket/ssh.handler.ts
    MODIFICARE: server/src/services/connection.service.ts

[ ] ZT-036 — Security headers (helmet middleware)
------------------------------------------------------------------------------
  Priorita: ALTA
  Dipendenze: Nessuna

  DESCRIZIONE:
  Aggiungere il middleware helmet per impostare gli header HTTP di sicurezza
  standard. Attualmente l'app non invia nessun header di sicurezza (CSP, HSTS,
  X-Frame-Options, X-Content-Type-Options, Referrer-Policy), esponendola a
  attacchi XSS, clickjacking e content sniffing.

  DETTAGLI TECNICI:
  Backend:
    - Installare helmet: npm install helmet
    - Aggiungere app.use(helmet({...})) in server/src/app.ts prima delle routes
    - Configurare Content-Security-Policy per permettere le risorse del client
    - Configurare HSTS con max-age appropriato (31536000s = 1 anno)
    - Abilitare X-Frame-Options: DENY (l'app non deve essere in iframe)
    - Abilitare X-Content-Type-Options: nosniff
    - Configurare Referrer-Policy: strict-origin-when-cross-origin
  File coinvolti:
    - server/src/app.ts
    - server/package.json

[ ] ZT-042 — Rotazione refresh token
------------------------------------------------------------------------------
  Priorita: MEDIA
  Dipendenze: Nessuna

  DESCRIZIONE:
  Implementare la rotazione dei refresh token: ogni volta che un refresh
  token viene usato, deve essere invalidato e sostituito con uno nuovo.
  Attualmente lo stesso refresh token resta valido per tutta la sua durata
  (7 giorni), e se rubato puo' essere usato indefinitamente.
  Aggiungere anche reuse detection: se un token gia' ruotato viene riutilizzato,
  revocare tutti i token della famiglia (indicatore di furto token).

  DETTAGLI TECNICI:
  Backend:
    - In auth.service.ts refreshAccessToken():
      1. Trovare il refresh token nel DB
      2. Cancellare il vecchio token
      3. Generare un nuovo refresh token (UUID)
      4. Salvare il nuovo token nel DB con stessa scadenza relativa
      5. Restituire sia accessToken che newRefreshToken nella response
    - Aggiungere campo tokenFamily String? al modello RefreshToken per
      raggruppare i token della stessa sessione
    - Reuse detection: se il token usato e' gia' stato ruotato (non esiste
      nel DB), cancellare TUTTI i token della stessa famiglia → forza
      re-login su tutti i dispositivi
  Frontend:
    - Aggiornare il client Axios interceptor per salvare il nuovo refresh
      token ricevuto nella response
  File coinvolti:
    - server/src/services/auth.service.ts (refreshAccessToken)
    - server/prisma/schema.prisma (RefreshToken model)
    - client/src/api/client.ts o equivalente (interceptor)
