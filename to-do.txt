[ ] SSHGW-050 — Generazione e gestione chiavi SSH server-side per gateway gestiti
------------------------------------------------------------------------------
  Priorita: ALTA
  Dipendenze: SSHGW-049

  DESCRIZIONE:
  Implementare la generazione e il ciclo di vita di una coppia di chiavi SSH
  (Ed25519) utilizzata dal server RDM per autenticarsi verso i gateway SSH
  gestiti. La chiave privata viene cifrata e salvata nel database con una
  chiave server-level (non la vault utente — il server deve poter decifrare
  autonomamente durante le connessioni). La chiave pubblica viene esposta
  tramite API per essere iniettata nei container gateway. Questo elimina la
  necessita' di gestire password per i gateway managed.

  DETTAGLI TECNICI:
  Prisma:
    - Nuovo modello SshKeyPair:
        id                   String   @id @default(uuid())
        tenantId             String   @unique
        encryptedPrivateKey  String
        privateKeyIV         String
        privateKeyTag        String
        publicKey            String   // chiave pubblica in chiaro
        fingerprint          String   // SHA256 fingerprint
        algorithm            String   @default("ed25519")
        createdAt            DateTime @default(now())
        updatedAt            DateTime @updatedAt
        tenant               Tenant   @relation(fields: [tenantId],
                                      references: [id], onDelete: Cascade)
    - Aggiungere relazione in Tenant: sshKeyPair SshKeyPair?
    - Migrazione Prisma
  Crittografia server-level:
    - Nuova env var: SERVER_ENCRYPTION_KEY (32 bytes hex)
    - In config.ts: aggiungere SERVER_ENCRYPTION_KEY (obbligatorio in
      production, auto-generato in development)
    - In crypto.service.ts: aggiungere encryptWithServerKey()/
      decryptWithServerKey() che usano SERVER_ENCRYPTION_KEY
    - La chiave privata SSH e' cifrata con questa key, NON con la
      vault utente — il server deve poter decifrare senza interazione
  Servizio (sshkey.service.ts):
    - generateKeyPair(tenantId):
        - Usare crypto.generateKeyPairSync('ed25519') di Node.js
        - Cifrare la chiave privata con encryptWithServerKey()
        - Salvare nel modello SshKeyPair
        - Restituire la chiave pubblica e il fingerprint
    - getPublicKey(tenantId):
        - Restituire solo la chiave pubblica (non sensibile)
    - getPrivateKey(tenantId):
        - Decifrare la chiave privata con decryptWithServerKey()
        - Restituire come Buffer per uso in ssh2 connect()
    - rotateKeyPair(tenantId):
        - Generare nuova coppia, sostituire la precedente
        - Restituire la nuova chiave pubblica (l'utente deve aggiornare
          i container gateway con la nuova authorized_keys)
  Controller e Routes:
    - POST  /api/gateways/ssh-keypair          — genera coppia di chiavi
    - GET   /api/gateways/ssh-keypair          — restituisce pubblica +
                                                  fingerprint
    - POST  /api/gateways/ssh-keypair/rotate   — ruota la coppia
    - Proteggere con requireTenantRole('ADMIN')
    - Zod validation, audit logging (nuovi AuditAction: SSH_KEY_GENERATE,
      SSH_KEY_ROTATE)
  File coinvolti:
    MODIFICARE: server/prisma/schema.prisma (SshKeyPair model, Tenant rel.)
    CREARE:     server/src/services/sshkey.service.ts

[ ] VAULT-058 — External sharing: link temporanei per condivisione esterna
------------------------------------------------------------------------------
  Priorita: MEDIA
  Dipendenze: VAULT-054

  DESCRIZIONE:
  Implementare la condivisione esterna di secrets tramite link temporanei.
  Un utente puo' generare un link per condividere un secret con persone
  che non hanno un account nel sistema. Il link ha scadenza temporale
  configurabile, numero massimo di accessi, e PIN opzionale.

  Modello crittografico: il token casuale (32 bytes) e' la chiave. Il DB
  salva solo SHA-256(token) per lookup. Senza PIN: chiave = HKDF(token).

          dataIV          String
          dataTag         String
          hasPin          Boolean   @default(false)
          pinSalt         String?             // salt per derivazione chiave con PIN
          expiresAt       DateTime
          maxAccessCount  Int?                // null = unlimited

          secretType      SecretType          // tipo del secret (per display)
          secretName      String              // nome del secret (non cifrato)

        expiresInHours, maxAccessCount?, pin? }, tenantId?):

        2. Generare token: crypto.randomBytes(32).toString('base64url')
        3. Calcolare tokenHash: SHA-256(token)
        4. Decifrare il secret con la chiave sorgente

        2. Cercare ExternalSecretShare by tokenHash
        3. Validare: non scaduto, non revocato, accessCount < maxAccessCount
        4. Derivare chiave (stessa logica di creazione)
        5. Decifrare i dati
        6. Incrementare accessCount
        7. Audit log SECRET_EXTERNAL_ACCESS (senza userId, con IP)
        8. Restituire dati decifrati
